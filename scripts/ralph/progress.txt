## Codebase Patterns
- Chat transcript and live assembler mutations should call `refreshViewportContentFollow(...)` or `refreshLiveViewportContent()` explicitly; do not rely on `View()` to rebuild persistent chat state.
- Live tool results that cannot reconcile to an existing tool cell still need an explicit `refreshLiveViewportContent()`; otherwise they stay invisible until the remaining pending tool calls finally commit.
- Composer edits should only refresh the chat viewport when `viewportBodyHeight(mainWidth)` changes; same-height edits should leave `chat.SetMessages()` untouched.
- When only the body height changes at the current width, reuse the existing `chatList` items and sync layout with `chat.SetSize(...)` plus offset clamp or tail scroll; reserve `chat.SetMessages()` for width, page, or content changes.
- Once tail-only rendering is active on the chat page, the first scroll-away action (`PgUp`, `Ctrl+U`, `Home`) must rebuild the full chat list before changing offsets; otherwise history navigation operates inside a cropped tail window.
- Long-history composer perf tests should seed hundreds of transcript messages and guard both `chat.generation` and `len(m.chat.items)`; use rune/backspace edit cycles in benchmarks so composer height stays stable and measurements isolate the normal edit path.
